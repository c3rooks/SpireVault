# Security & Trust Model

The Vault is open source on purpose. Read the code, run it through your own
linters and audit tools, fork it if you don't trust the hosted server.

This document describes what we defend against, what we *don't* defend
against, and what we'd actually classify as a security bug.

If you find something not covered here, please follow [Reporting a
vulnerability](#reporting-a-vulnerability) below — please don't open a
public issue.

---

## What can a malicious actor reach?

Three surfaces, ordered by accessibility:

1. **The presence Worker** at `vault-coop.coreycrooks.workers.dev`.
   Public HTTP endpoints. CORS is open. Anyone can hit it.
2. **The web companion** at `app.spirevault.app`.
   Static HTML/CSS/JS hosted on Cloudflare Pages. Same-origin to the user.
3. **The macOS app** running locally on a user's Mac.
   No inbound network surface. Only outbound to the Worker.

We design defenses around the assumption that someone is *actively trying*
to exploit each of those.

## Threat model

### Identity

| Concern | Defense |
| --- | --- |
| Can I post presence as another player? | **No.** `POST /presence` requires `Authorization: Bearer <session>`. The Worker resolves the token to a verified SteamID server-side. The body's SteamID is ignored — the server stamps its own. |
| Can I forge a session token? | **No.** Tokens are 24 random bytes generated by `crypto.getRandomValues`. They're stored only in KV — there's no signature to forge. Guessing a valid token is computationally infeasible. |
| Can I steal a session token? | Only if you can read someone's local files (macOS app: `~/Library/Application Support/.../steam-session.json`) or their browser localStorage. Both require already-compromised access to the device. Tokens travel only over HTTPS to the Worker. |
| Can I impersonate another player's name/avatar? | **No.** Persona name and avatar URL are pulled from Steam Web API at OpenID time and pinned server-side. The client cannot supply these fields in `POST /presence` — they're overwritten. |
| Can a malicious sign-in link steal my session? | **No.** The Worker's `safeReturnURL` allowlist restricts where OpenID can redirect with a session token (`thevault://`, `app.spirevault.app`, localhost, or env-allowlisted hosts). Anything else is silently downgraded. |

### Replay & MITM

| Concern | Defense |
| --- | --- |
| Replay of a successful sign-in URL? | The client generates a 16-byte nonce before sign-in and verifies it on return. A second visit with the same URL is rejected. |
| MITM of the OpenID flow? | The OpenID round-trip is HTTPS end-to-end; Steam validates the verification request server-side. The Worker cannot mint a session for a SteamID it didn't see Steam confirm. |
| MITM of the heartbeat? | All traffic is HTTPS to a Cloudflare-managed cert. There is no protocol downgrade (HSTS via `_headers`). |

### Abuse

| Concern | Defense |
| --- | --- |
| Spam by a logged-in user (heartbeat flood) | `MIN_HEARTBEAT_INTERVAL_MS = 2000` enforces 2 s spacing per session at the Worker. KV writes are also globally rate-limited by Cloudflare (1/sec/key). |
| Slur or PII in the public note field | Server-side `sanitizeNote` strips control chars and caps at 140 chars. We do not run a profanity filter — community moderation is a human problem. |
| Mass account creation to flood the feed | One Steam account = one feed row (the KV key is `presence:<steamID>`). Generating Steam accounts is not free of effort. |
| DDoS | Cloudflare's edge handles inbound DDoS for free. The Worker has zero compute cost when not running. |

### XSS / injection

| Concern | Defense |
| --- | --- |
| User notes/Discord rendered as HTML | Web client escapes `& < > "` in every user-controlled string before insertion. The macOS app uses SwiftUI `Text` which is HTML-immune. |
| Avatar URL with `javascript:` scheme | Web client validates the URL parses and uses `http`/`https` before rendering. Falls back to the bundled logo otherwise. |
| `steam://` deep links carrying injected query strings | Constructed only from a 17-digit-validated SteamID. The remainder of the URL is hard-coded. |
| CSP escapes | `Content-Security-Policy` in `_headers` for both Site and Web restricts script/style/connect/font origins explicitly. No `unsafe-eval`. |

### Privacy

| Concern | Defense |
| --- | --- |
| What does the server store about me? | Verified SteamID, persona, avatar URL (all already public on Steam), plus your status / note / Discord handle (all your input). 30-day session token. Nothing else. |
| What does the macOS app upload? | Only what you typed into the Co-op tab. Run history, decks, save files **never** leave your Mac. |
| Can someone scrape my Discord handle? | Only if you put it in the public note. The web companion and macOS app both make this opt-in. The handle is exposed to anyone who reads the public `/presence` feed; that's the point. |
| What gets logged? | Cloudflare Workers logs (request method, path, status) for 7 days. We do not log user-agent strings or bodies. There is no analytics pipeline. |

## What we deliberately **don't** defend against

Out of scope. Pretending we'd defend against these would be lying.

- **A user who shares their session token publicly.** That's revealing a
  credential they own. Sign out (drops the row, expires the token).
- **A user who pastes their own Discord handle and complains it leaked.**
  That's a privacy choice the user actively made.
- **Someone running a fork of the Worker on their own domain.** They can do
  whatever they want with their own users; that's why it's open source.
  The default Vault build only talks to our official Worker URL.
- **Steam compromising your Steam account.** Out of scope; that's between
  you and Valve. We can't help once your password is stolen.
- **OS-level malware on your Mac reading your Application Support folder.**
  If you're already pwned, this is the least of your problems.

## "Is the GitHub repo safe to clone?"

Yes. Specifically:

- **The macOS app** has no remote code execution path. It does not download
  or evaluate code from the internet. It does not auto-update. (Sparkle and
  similar update frameworks are deliberately not used.)
- **It only sends outbound traffic to** the Worker URL configured at compile
  time (`AppConfig.defaultServerURL`) plus, transitively through Steam
  OpenID, `steamcommunity.com`. Zero analytics, zero telemetry, no other
  hosts. You can verify this with Little Snitch or `tcpdump`.
- **The Backend** is ~400 lines of TypeScript. There's no native code, no
  C-bindings, no `eval`. Every dependency is in `package.json`; only
  `wrangler` and `typescript` are installed (both Cloudflare/Microsoft
  upstream).
- **The Site and Web are pure static** — no build step, no Node runtime in
  production, no node_modules served to browsers.

If something *did* end up malicious — say, a contributor sneaks an
exfiltrating fetch into a PR — `git log` would show it, `_headers` CSP would
block the fetch in the Web client, and a normal `make run` of the macOS app
would be observable on `tcpdump`.

## Reporting a vulnerability

Email the maintainer at the address listed on
<https://github.com/c3rooks>. Include reproduction steps and impact.

I'll respond within a few days, fix and deploy as fast as I can, and credit
you publicly in the release notes if you'd like.

Please **don't** open a public GitHub issue for security-impacting bugs;
file a regular issue once a fix has shipped.
